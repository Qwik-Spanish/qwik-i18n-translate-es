# Qwik en resumen

## Componentes

Los componentes de Qwik son muy similares a los componentes de React. Son funciones que devuelven JSX. Sin embargo, se debe utilizar `component$(...)`, los controladores de eventos deben tener el sufijo `$`, se crea estado utilizando `useSignal()`, se utiliza `class` en lugar de `className` y hay algunas otras diferencias.

- Los componentes siempre se declaran con la funci√≥n `component$`.
- Los componentes pueden utilizar el hook `useSignal` para crear estado reactivo.
- Los controladores de eventos se declaran con el sufijo `$`.
- Para `<input>`, el evento `onChange` se llama `onInput$` en Qwik.
- JSX prefiere los atributos HTML. `class` en lugar de `className`. `for` en lugar de `htmlFor`.
- El renderizado condicional se realiza con el operador ternario `?` y el operador `&&`, al igual que en React.

```tsx title="src/components/my-other-component/index.tsx"
import { component$, Slot } from '@builder.io/qwik';
import type { ClassList } from '@builder.io/qwik'

export component$((props: { class?: ClassList }) => { // ‚úÖ
  return <div class={class}><Slot /></div>;
});
```

```tsx
import { component$, useSignal } from '@builder.io/qwik';

// Otros componentes se pueden importar y usar en JSX.
import { MyOtherComponent } from './my-other-component';

interface MyComponentProps {
  step: number;
}

// Los componentes siempre se declaran con la funci√≥n `component$`.
export const MyComponent = component$((props: MyComponentProps) => {
  // Los componentes utilizan el hook `useSignal` para crear estado reactivo.
  const count = useSignal(0); // { value: 0 }
  return (
    <>
      <button
        onClick$={() => {
          // Los controladores de eventos tienen el sufijo `$`.
          count.value = count.value + props.step;
        }}
      >
        Incrementar en {props.step}
      </button>
      <main
        class={{
          even: count.value % 2 === 0,
          odd: count.value % 2 === 1,
        }}
      >
        <h1>Conteo: {count.value}</h1>
        <MyOtherComponent class="forma-correcta" /> {/* ‚úÖ */}
        {count.value > 10 && <p>El conteo es mayor que 10</p>}
        {count.value > 10 ? <p>El conteo es mayor que 10</p> : <p>El conteo es menor que 10</p>}
      </main>
    </>
  );
});
```


### Reglas de uso de hooks

Los m√©todos que comienzan con `use` son especiales en Qwik, como `useSignal()`, `useStore()`, `useOn()`, `useTask$()`, `useLocation()` y otros, muy similares a los hooks de React.

- Solo pueden ser llamados dentro de un componente `component$`.
- Solo pueden ser llamados desde el nivel superior de un componente `component$`, no dentro de una estructura condicional o un bucle.

## Estilizaci√≥n

Qwik admite de manera nativa CSS modules, Tailwind CSS, importaci√≥n global de CSS y carga diferida de CSS con alcance utilizando `useStyleScoped$()`. Los CSS modules son la forma recomendada de estilizar los componentes de Qwik.

### CSS Modules

Para usar CSS modules, simplemente crea un archivo `.module.css`. Por ejemplo, `src/components/MyComponent/MyComponent.module.css`.

```css title="src/components/MyComponent/MyComponent.module.css"
.container {
  background-color: red;
}
```

Luego, importa el m√≥dulo CSS en tu componente.

```tsx title="src/components/MyComponent/MyComponent.tsx"

  return <div class={styles.container}>Hola mundo</div>;

```

Recuerda que Qwik utiliza `class` en lugar de `className` para las clases CSS.

## Reglas de $(...)

La funci√≥n `$(...)` y cualquier funci√≥n que termine con `$` son especiales en Qwik, como `$()`, `useTask$()`, `useVisibleTask$()` y otros. El s√≠mbolo `$` al final representa un l√≠mite de carga diferida (lazy loading). Existen algunas reglas que se aplican al primer argumento de cualquier funci√≥n `$`. No tiene ninguna relaci√≥n con jQuery.

- El primer argumento debe ser una variable importada.
- El primer argumento debe ser una variable declarada en el nivel superior del mismo m√≥dulo.
- El primer argumento debe ser una expresi√≥n que contenga variables.
- Si el primer argumento es una funci√≥n, solo puede capturar variables que est√©n declaradas en el nivel superior del mismo m√≥dulo o cuyo valor sea serializable. Los valores serializables incluyen: `string`, `number`, `boolean`, `null`, `undefined`, `Array`, `Object`, `Date`, `RegExp`, `Map`, `Set`, `BigInt`, `Promise`, `Error`, `nodos JSX`, `Signal`, `Store` e incluso `HTMLElements`.

```tsx

  console.log('funci√≥n exportada');

  // El primer argumento es una funci√≥n.
  const valido1 = $((event) => {
    console.log('input', event.target.name, event.target.value);
  });

  // El primer argumento es un identificador importado.
  const valido2 = $(importedFunction);

  // El primer argumento es un identificador declarado en el nivel superior del mismo m√≥dulo.
  const valido3 = $(exportedFunction);

  // El primer argumento es una expresi√≥n sin variables locales.
  const valido4 = $([1, 2, { a: 'hola' }]);

  // El primer argumento es una funci√≥n que captura una variable local.
  const variableLocal = 1;
  const valido5 = $((event) => {
    console.log('variable local', variableLocal);
  });
});
```

Aqu√≠ hay algunos ejemplos de funciones `$` inv√°lidas.

```tsx

```
## Estado reactivo

### `useSignal(initialValue?)`

`useSignal()` es la forma principal de crear un estado reactivo. Las se√±ales (signals) pueden ser compartidas entre componentes, y cualquier componente o tarea que lea la se√±al (ejecutando: `signal.value`) se renderizar√° cuando la se√±al cambie.

```ts
// Definici√≥n de TypeScript para `Signal<T>` y `useSignal<T>`

export interface Signal<T> {
  value: T;
}

export const useSignal: <T>(value?: T | (() => T)): Signal<T>;
```

`useSignal(initialValue?)` toma un valor inicial opcional y devuelve un objeto `Signal<T>`. El objeto `Signal<T>` tiene una propiedad `value` que puede ser le√≠da y escrita. Cuando un componente o tarea accede a la propiedad `value`, se crea autom√°ticamente una suscripci√≥n, por lo que cuando el `value` cambia, todos los componentes, tareas u otras se√±ales computadas que lean el `value` se reevaluar√°n.

### `useStore(initialValue?)`

`useStore(initialValue?)` es similar a `useSignal`, excepto que crea un objeto javascript reactivo, haciendo que cada propiedad del objeto sea reactiva, al igual que el `value` de una se√±al. En la implementaci√≥n, `useStore` utiliza un objeto `Proxy` que intercepta todos los accesos a las propiedades, haciendo que la propiedad sea reactiva.

```ts
// Definici√≥n de TypeScript para `useStore<T>`

// `Reactive<T>` es una versi√≥n reactiva del tipo `T`, cada propiedad de `T` se comporta como una `Signal<T>`.
export interface Reactive<T extends Record<string, any>> extends T {}

export interface StoreOptions {
  // Si `deep` es true, entonces las propiedades anidadas del store se envolver√°n en una `Signal<T>`.
  deep?: boolean;
}
export const useStore: <T>(value?: T | (() => T), options?: StoreOptions): Reactive<T>;
```

En la pr√°ctica, `useSignal` y `useStore` son muy similares -- `useSignal(0) === useStore({ value: 0 })` -- pero `useSignal` es preferible la mayor√≠a de las veces. Algunos casos de uso para `useStore` son:

- Cuando se necesita reactividad en un array.
- Cuando se desea un objeto reactivo al que se puedan agregar f√°cilmente propiedades.
```tsx
import { component$, useSignal } from '@builder.io/qwik';

export const Counter = component$(() => {
  // El hook `useStore` se utiliza para crear un almac√©n reactivo.
  const todoList = useStore(
    {
      array: [],
    },
    { deep: true }
  );

  // `todoList.array` es un array reactivo, por lo que podemos agregar elementos y el componente se volver√° a renderizar.

  return (
    <>
      <h1>Lista de tareas</h1>
      <ul>
        {todoList.array.map((todo) => (
          <li key={todo.id}>
            <input
              type="checkbox"
              checked={todo.completed}
              onInput$={() => {
                // `todoList` es un almac√©n reactivo
                // debido a que usamos `deep: true`, el objeto `todo` tambi√©n es reactivo.
                // por lo tanto, podemos cambiar la propiedad `completed` y el componente se volver√° a renderizar.
                todo.completed = !todo.completed;
              }}
            />
            {todo.text}
          </li>
        ))}
      </ul>
    </>
  );
});
```

### `useTask$(() => { ... })`

`useTask$` se utiliza para crear una tarea as√≠ncrona. Las tareas son √∫tiles para implementar efectos secundarios, ejecutar c√°lculos intensivos y c√≥digo as√≠ncrono como parte del ciclo de renderizado. Las tareas creadas con `useTask$` se ejecutan antes del primer renderizado y posteriormente cada vez que una se√±al o un almac√©n rastreado cambia.

```tsx

  const currentPage = useSignal(0);
  const listOfUsers = useSignal([]);

  // El hook `useTask$` se utiliza para crear una tarea.
  useTask$(() => {
    // La tarea se ejecuta antes del primer renderizado.
    console.log('Tarea ejecutada antes del primer renderizado');
  });

  // Puedes crear m√∫ltiples tareas y pueden ser as√≠ncronas.
  useTask$(async (taskContext) => {
    // Dado que queremos volver a ejecutar la tarea cuando cambie `currentPage`, 
    // debemos rastrearlo.
    taskContext.track(() => currentPage.value);
    console.log('Tarea ejecutada antes del primer renderizado Y cuando cambia la p√°gina');
    console.log('P√°gina actual:', currentPage.value);

    // Las tareas pueden ejecutar c√≥digo as√≠ncrono, como recuperar datos.
    const res = await fetch(`https://api.randomuser.me/?page=${currentPage.value}`);
    const json = await res.json();

    // Asignar a una se√±al provocar√° un nuevo renderizado.
    listOfUsers.value = json.results;
  });

  return (
    <>
      <h1>P√°gina {currentPage.value}</h1>
      <ul>
        {listOfUsers.value.map((user) => (
          <li key={user.login.uuid}>
            {user.name.first} {user.name.last}
          </li>
        ))}
      </ul>
      <button onClick$={() => currentPage.value++}>Siguiente p√°gina</button>
    </>
  );
});
```

`useTask$()` se ejecutar√° en el servidor durante el renderizado del servidor (SSR) y en el navegador si el componente se monta por primera vez en el cliente. Debido a esto, no es una buena idea acceder a las APIs del DOM en una tarea, ya que no estar√°n disponibles en el servidor. En su lugar, debes utilizar controladores de eventos o `useVisibleTask$()` para ejecutar una tarea solo en el cliente/navegador.

`useVisibleTask$(() => { ... })` se utiliza para crear una tarea que ocurre justo DESPU√âS de que el componente se monta por primera vez en el DOM. Es similar a `useTask$`, excepto que solo se ejecuta en el cliente y se ejecuta despu√©s del primer renderizado. Debido a que se ejecuta despu√©s del renderizado, est√° permitido inspeccionar el DOM o utilizar las API del navegador.

```tsx
import { component$, useSignal, useVisibleTask$ } from '@builder.io/qwik';

export const Reloj = component$(() => {
  const tiempo = useSignal(0);

  // El hook `useVisibleTask$` se utiliza para crear una tarea que se ejecuta de forma activa en el cliente.
  useVisibleTask$((taskContext) => {
    // Dado que esta TareaVisible no est√° rastreando ninguna se√±al, solo se ejecutar√° una vez.

    const intervalo = setInterval(() => {
      tiempo.value = new Date();
    }, 1000);

    // La funci√≥n `cleanup` se llama cuando el componente se desmonta o cuando la tarea se vuelve a ejecutar.
    taskContext.cleanup(() => clearInterval(intervalo));
  });

  return (
    <>
      <h1>Reloj</h1>
      <h1>Segundos transcurridos: {tiempo.value}</h1>
    </>
  );
});
```

Dado que Qwik NO ejecuta ning√∫n JavaScript en el navegador antes de la interacci√≥n del usuario, `useVisibleTask$()` es la √∫nica API que se ejecutar√° de forma activa en el cliente. Por eso es un buen lugar para realizar acciones como:

- Acceder a las API del DOM.
- Inicializar bibliotecas espec√≠ficas del navegador.
- Ejecutar c√≥digo de an√°lisis.
- Iniciar una animaci√≥n o un temporizador.

Ten en cuenta que `useVisibleTask$()` no debe utilizarse para obtener datos, ya que no se ejecutar√° en el servidor. En su lugar, debes utilizar `useTask$()` para obtener datos y luego utilizar `useVisibleTask$()` para realizar acciones como iniciar una animaci√≥n. Abusar de `useVisibleTask$()` puede afectar el rendimiento.

## Enrutamiento

Qwik viene con un enrutador basado en archivos, similar a Next.js, pero con algunas diferencias. El enrutador se basa en el sistema de archivos, espec√≠ficamente en `src/routes/`. Al crear un nuevo archivo `index.tsx` en una carpeta dentro de `src/routes/`, se crea una nueva ruta. Por ejemplo, `src/routes/home/index.tsx` crear√° una ruta en `/home/`.

```tsx title="src/routes/home/index.tsx"

import { component$ } from '@builder.io/qwik';

export default component$(() => {
  return <h1>Inicio</h1>;
});
```

Es muy importante exportar el componente como la exportaci√≥n predeterminada, de lo contrario, el enrutador no podr√° encontrarlo.

Puedes crear rutas din√°micas agregando una carpeta con `[parametro]` en la ruta. Por ejemplo, `src/routes/user/[id]/index.tsx` crear√° una ruta en `/user/:id/`. Para acceder al par√°metro de la ruta, puedes utilizar el hook `useLocation` exportado de `@builder.io/qwik-city`.

```tsx title="src/routes/user/[id]/index.tsx"

import { component$ } from '@builder.io/qwik';
import { useLocation } from '@builder.io/qwik-city';

export default component$(() => {
  const loc = useLocation();
  return (
    <main>
      {loc.isNavigating && <p>Cargando...</p>}
      <h1>Usuario: {loc.params.id}</h1>
      <p>URL actual: {loc.url.href}</p>
    </main>
  );
});
```

`useLocation()` devuelve un objeto reactivo `RouteLocation`, lo que significa que se volver√° a renderizar cada vez que cambie la ruta. El objeto `RouteLocation` tiene las siguientes propiedades:

```tsx
/**
 * La ubicaci√≥n de ruta actual devuelta por `useLocation()`.
 */
export interface RouteLocation {
  readonly params: Readonly<Record<string, string>>;
  readonly url: URL;
  readonly isNavigating: boolean;
}
```

### Enlazar a otras rutas

Para enlazar a otras rutas, puedes utilizar el componente `Link` exportado de `@builder.io/qwik-city`. El componente `Link` toma todas las propiedades del elemento `<a>` HTMLAnchorElement. La √∫nica diferencia es que utilizar√° el enrutador de Qwik para navegar a la ruta en una SPA en lugar de realizar una navegaci√≥n completa de la p√°gina.

La forma recomendada de cargar datos desde el servidor es utilizar la funci√≥n `routeLoader$()` exportada de `@builder.io/qwik-city`. La funci√≥n `routeLoader$()` se utiliza para crear un cargador de datos que se ejecutar√° en el servidor antes de que se renderice la ruta. El resultado de `routeLoader$()` debe exportarse como una exportaci√≥n con nombre desde el archivo de ruta, es decir, solo se puede utilizar en un archivo `index.tsx` dentro de `src/routes/`.

```tsx title="src/routes/user/[userID]/index.tsx"

// El resultado de `routeLoader$()` es un hook personalizado que se puede utilizar para acceder a los datos devueltos por `routeLoader$()`.

  const user = await db.table('users').get(requestContext.params.userID);

    name: user.name,
    email: user.email,

  // El hook `useUserData` devolver√° una `Signal` que contiene los datos devueltos por `routeLoader$()`, lo que volver√° a renderizar el componente cuando cambie la navegaci√≥n y se vuelva a ejecutar `routeLoader$()`.
  const userData = useUserData();
  return (
    <main>
      <h1>Datos del usuario</h1>
      <p>Nombre de usuario: {userData.value.name}</p>

    title: `Usuario: "${user.name}"`,

        name: 'description',

```

La funci√≥n `routeLoader$()` toma una funci√≥n que devuelve una promesa. La promesa se resuelve en el servidor y el valor resuelto se pasa al hook `useCustomLoader$()`. El hook `useCustomLoader$()` es un hook personalizado que se crea con la funci√≥n `routeLoader$()`. El hook `useCustomLoader$()` devuelve una `Signal` que contiene el valor resuelto de la promesa devuelta por la funci√≥n `routeLoader$()`. El hook `useCustomLoader$()` volver√° a renderizar el componente cuando cambie la ruta y se vuelva a ejecutar la funci√≥n `routeLoader$()`.

Qwik proporciona la API `routeAction$()` exportada desde `@builder.io/qwik-city` para manejar las solicitudes de formularios en el servidor. `routeAction$()` solo se ejecuta en el servidor cuando se env√≠a el formulario.

```tsx title="src/routes/user/[userID]/index.tsx"

  user.email = data.email;
  await db.table('users').put(user);
  return {
    user,
  };
}, zod$({
  name: z.string(),
  email: z.string(),
}));

  // El hook `useUserUpdate` devolver√° un `ActionStore<T>` que contiene el `value` devuelto por `routeAction$()`, y otras propiedades, como `submit()`, que se utiliza para enviar el formulario program√°ticamente, y `isRunning`.   const userData = useUserUpdate();
  // userData.value es el valor devuelto por `routeAction$()`, que es `undefined` antes de enviar el formulario.
  // userData.formData es los datos del formulario que se enviaron, es `undefined` antes de enviar el formulario.
  // userData.isRunning es un booleano que es verdadero cuando se est√° enviando el formulario.
  // userData.submit() es una funci√≥n que se puede utilizar para enviar el formulario program√°ticamente.
  // userData.actionPath es la ruta a la acci√≥n, que se utiliza para enviar el formulario.
  return (
    <main>
      <h1>Datos del usuario</h1>

        </div>
        <div>
          <label>Email de usuario: <input name="email" defaultValue={userData.formData?.get('email')} /></label>

```

`routeAction$()` se utiliza junto con el componente `Form` exportado desde `@builder.io/qwik-city`. El componente `Form` es un contenedor alrededor del elemento HTML nativo `<form>`. El componente `Form` toma un `ActionStore<T>` como propiedad `action`. El `ActionStore<T>` es el valor de retorno de la funci√≥n `routeAction$()`.

## Ejecutar el c√≥digo √∫nicamente en el navegador

Dado que Qwik ejecuta el mismo c√≥digo en el servidor y en el navegador, no puedes usar `window` u otras API del navegador en tu c√≥digo, porque no existen cuando el c√≥digo se ejecuta en el servidor.

Si deseas acceder a las API del navegador, como `window`, `document`, `localStorage`, `sessionStorage`, `webgl`, etc., debes verificar si el c√≥digo se est√° ejecutando en el navegador antes de acceder a las API del navegador.

```tsx
import { component$, useTask$, useVisibleTask$, useSignal } from '@builder.io/qwik';
import { isBrowser } from '@builder.io/qwik/build';

export default component$(() => {
  const ref = useSignal<Element>();

  // useVisibleTask$ se ejecutar√° solo en el navegador
  useVisibleTask$(() => {
    // No es necesario verificar `isBrowser` antes de acceder al DOM, porque useVisibleTask$ se ejecutar√° solo en el navegador
    ref.value?.focus();
    document.title = 'Hola mundo';
  });

  // useTask puede ejecutarse en el servidor, por lo que debes verificar `isBrowser` antes de acceder al DOM
  useTask$(() => {
    if (isBrowser) {
      // Este c√≥digo se ejecutar√° solo en el navegador cuando el componente se renderice all√≠ por primera vez
      ref.value?.focus();
      document.title = 'Hola mundo';
    }
  });

  return (
    <button
      ref={ref}
      onClick$={() => {
        // Todos los controladores de eventos se ejecutan solo en el navegador, por lo que es seguro acceder al DOM
        ref.value?.focus();
        document.title = 'Hola mundo';
      }}
    >
      Haz click aqu√≠
    </button>
  );
});
```
### `useVisibleTask$(() => { ... })`

Esta API declara una VisibleTask que se asegura de que se ejecute solo en el cliente/navegador. Nunca se ejecutar√° en el servidor.

### Manejadores de eventos JSX

Los manejadores JSX, como `onClick$` y `onInput$`, solo se ejecutan en el cliente. Esto se debe a que son eventos del DOM, y dado que no hay DOM en el servidor, no se ejecutar√°n en el servidor.

### Condicional `isBrowser`

Qwik proporciona un booleano `isBrowser` exportado desde `@builder.io/qwik/build`. Puedes usar esto para asegurarte de que el c√≥digo se ejecute solo en el navegador.

### `routeAction$()`

`routeAction$()` es un componente especial que se ejecuta solo en el servidor. Se utiliza para manejar env√≠os de formularios y otras acciones. Por ejemplo, se puede utilizar para agregar un usuario a una base de datos y luego redirigir a la p√°gina de perfil del usuario.

### `routeLoader$()`

`routeLoader$()` es un componente especial que se ejecuta solo en el servidor. Se utiliza para obtener datos y luego renderizar la p√°gina. Por ejemplo, se puede utilizar para obtener datos de una API y luego renderizar la p√°gina con los datos.

### `server$((...args) => { ... })`

`server$()` es una forma especial de declarar funciones que solo se ejecutan en el servidor. Si se llaman desde el cliente, se comportar√°n como una llamada RPC y se ejecutar√°n en el servidor. Pueden tomar cualquier argumento serializable y devolver cualquier valor serializable.

### El condicional `isServer`

Qwik proporciona un booleano `isServer` exportado desde `@builder.io/qwik/build`. Puedes usar esto para asegurarte de que el c√≥digo se ejecute solo en el servidor.
